# UI Bugy — 2026-02-26 (odpoledne)

## BUG 1: Notifikační dialog — tlačítka "Zahodit" / "Zkusit znovu" nereagují

**Priorita**: HIGH
**Screenshoty**: dialog "Zpracování odkazu selhalo" s tlačítky Zahodit a Zkusit znovu

### Problém

Kliknutí na "Zahodit" nebo "Zkusit znovu" v error notifikaci vizuálně nic neudělá.
Dialog zůstane otevřený, žádná zpětná vazba uživateli.

### Root cause

`NotificationViewModel.kt` řádky 131-160:

```kotlin
fun discardTask(taskId: String, onError: (String) -> Unit = {}) {
    scope.launch {
        try {
            repository.userTasks.cancel(taskId)   // ← TADY selže (Spring Data bug)
            _userTaskDialogEvent.value = null      // ← nikdy se neprovede
            ...
        } catch (e: Exception) {
            onError("Zahození selhalo: ${e.message}")  // ← chyba jde sem, ale dialog zůstane
        }
    }
}
```

Tlačítko FUNGUJE — volá backend, backend SELŽE (Spring Data mapping error),
výjimka se zachytí, `onError` se zavolá, ale **dialog se nezavře**.
Uživatel nevidí žádnou zpětnou vazbu.

### Řešení

1. **Dialog musí mít dismiss i při chybě** — přidat `_userTaskDialogEvent.value = null`
   do catch bloku (nebo `finally`), aby se dialog zavřel i při selhání
2. **Nebo zobrazit chybu přímo v dialogu** — přidat loading/error stav do dialogu
3. **Hlavně opravit BUG 2** (Spring Data) — to je root cause proč backend volání selhávají

### Soubory

- `shared/ui-common/.../notification/NotificationViewModel.kt` — řádky 131-160
- `shared/ui-common/.../notification/ApprovalNotificationDialog.kt` — řádky 95-171


## BUG 2: Spring Data "Parameter does not have a name" — STÁLE PŘETRVÁVÁ

**Priorita**: CRITICAL
**Screenshot**: macOS error dialog v User Tasks — "Parameter org.springframework.data.mapping.Parameter@a0e0341f does not have a name"

### Problém

Custom MongoDB convertery (`MongoValueClassConverters.kt`) **NEOPRAVILY root cause**.
Convertery řeší jen property-level konverzi (ObjectId ↔ TaskId), ale Spring Data
stále selhává při **constructor-level instantiation** TaskDocument.

### Proč convertery nestačí

Spring Data flow:
1. Čte document z MongoDB → má raw BSON (ObjectId, String, ...)
2. Hledá constructor `TaskDocument(...)` → reflektuje parameter names
3. Kotlin inline value class mangluje parameter names v bytecode
   (`id` → `id-<hash>`, `clientId` → `clientId-<hash>`)
4. Spring Data nenajde parameter name → **CRASH**
5. Custom convertery se použijí až v kroku PO construction — ale sem se nikdy nedostane

### Postižené služby

- `TaskQualificationService` — qualification loop padá každých 30s
- `BackgroundEngine` — nemůže číst tasky
- User Tasks UI — prázdná obrazovka + error dialog
- Notifikace — "Zahodit"/"Zkusit znovu" nefunguje (backend volání selhává)

### Správné řešení (jedno z):

**A) `@PersistenceCreator` s primitivními typy (doporučeno)**:
```kotlin
@Document(collection = "tasks")
data class TaskDocument(
    @Id val id: TaskId = TaskId.generate(),
    val clientId: ClientId,
    val projectId: ProjectId? = null,
    val sourceUrn: SourceUrn,
    // ... rest
) {
    companion object {
        @org.springframework.data.annotation.PersistenceCreator
        @JvmStatic
        fun create(
            id: org.bson.types.ObjectId,
            clientId: org.bson.types.ObjectId,
            projectId: org.bson.types.ObjectId?,
            sourceUrn: String,
            // ... ALL other constructor params with primitive types
        ): TaskDocument = TaskDocument(
            id = TaskId(id),
            clientId = ClientId(clientId),
            projectId = projectId?.let { ProjectId(it) },
            sourceUrn = SourceUrn(sourceUrn),
            // ...
        )
    }
}
```

**B) Nepoužívat value classes v entity (jednodušší)**:
```kotlin
@Document(collection = "tasks")
data class TaskDocument(
    @Id val id: ObjectId = ObjectId.get(),
    val clientId: ObjectId,
    val projectId: ObjectId? = null,
    val sourceUrn: String,
    // ...
) {
    fun taskId() = TaskId(id)
    fun clientId() = ClientId(clientId)
    // ... convenience methods
}
```

**C) Kotlin compiler flag** — ověřit že `-parameters` je v Gradle:
```kotlin
tasks.withType<KotlinCompile> {
    compilerOptions {
        javaParameters.set(true)  // POZOR: toto NEFUNGUJE pro inline value classes
    }
}
```
POZNÁMKA: `-parameters` flag nefunguje pro value classes protože mangling
je na úrovni Kotlin compileru, ne Java bytecode.

### Soubory

- `backend/server/.../entity/TaskDocument.kt` — hlavní entity
- `backend/server/.../configuration/MongoValueClassConverters.kt` — nefungující fix
- `backend/server/.../entity/FilteringRuleDocument.kt` — může mít stejný problém
- Všechny entity s value class fields v constructoru


## BUG 3: K8s namespace "bms-commerzbank" not found

**Priorita**: MEDIUM
**Screenshot**: Správa prostředí → BMS-Commerzbank → Error dialog

### Problém

Při práci s prostředím BMS-Commerzbank se Správa prostředí pokouší PATCHnout
ConfigMap v namespace `bms-commerzbank`, ale tento namespace neexistuje.

```
Failure executing: PATCH at:
https://10.96.0.1:443/api/v1/namespaces/bms-commerzbank/configmaps/DB-config?fieldManager=fabric8
Message: namespaces "bms-commerzbank" not found
```

### Root cause

Prostředí bylo vytvořeno v UI, ale namespace v K8s cluster nebyl vytvořen.
Buď:
- Environment setup neproběhl kompletně (namespace creation step selhal)
- Nebo namespace byl smazán ručně

### Řešení

1. **Ošetřit 404** — při pokusu o PATCH ConfigMap v neexistujícím namespace
   zobrazit srozumitelnou chybu ("Namespace neexistuje, vytvořte prostředí nejprve")
2. **Auto-create namespace** — pokud namespace neexistuje, vytvořit ho automaticky
   před PATCHem (záleží na business logice)
3. **Zkontrolovat environment setup flow** — proč se namespace nevytvořil

### Soubory

- Backend K8s client pro environment management (pravděpodobně `EnvironmentService.kt`)

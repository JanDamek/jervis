## 3. Problémy k řešení

### 3.1 Tier escalation local_large → local_xlarge (NEJHORŠÍ PROBLÉM)

**Fakta:**
- `select_local_tier()` v `provider.py:201-206`: `if context_tokens > 40_000: return LOCAL_XLARGE`
- LOCAL_LARGE num_ctx = 40,960 (viz `provider.py:64`)
- LOCAL_XLARGE num_ctx = 131,072 (viz `provider.py:69`)
- Gap: žádný tier mezi 40k a 131k
- Iter 1-4: estimated_tokens 39,014–39,907 → pod 40k → local_large → num_ctx=40,960 → OK
- Iter 5: estimated_tokens=40,731 → **překročí 40,000** → local_xlarge → num_ctx=**131,072** → katastrofa
- estimated_tokens roste s každou iterací: tool results (assistant msg + tool result msg) přidají ~200-800 tokenů

**Proč je to horší než před fixem:**
Před fixem LOCAL_LARGE = 49,152. estimated_tokens ~39-40k nikdy nepřekročily boundary 48k → zůstaly v local_large. Teď se boundary snížila na 40k spolu s num_ctx.

**Dopad:** Iter 5-6 + final streaming běží s 131k kontextem na P40 = masivní CPU RAM spill:
- Iter 5: 404s (vs ~200s na 40k)
- Iter 6: 453s
- Final streaming: **630s** pro 387 tokenů!

**VRAM aritmetika pro P40 (24GB):**
- 30b model weights (q4_K_M): ~17GB
- 40k KV cache: ~4GB → celkem ~21GB → **vejde se do 24GB** ✓
- 131k KV cache: ~13GB → celkem ~30GB → **6GB spill do CPU RAM** ✗
- S embedding modelem (8b = ~4.5GB): 17+4.5=21.5GB → 40k KV=4GB → 25.5GB → **1.5GB spill**

### 3.2 Model opakuje kb_search se stejným/podobným query

**Fakta:**
- Iter 1: `kb_search("systematic gaps in tracing data due to race condition")`
- Iter 4: `kb_search("systematic gaps in tracing data due to race condition")` — **identický** query
- Iter 5: `kb_search("race condition in session tracing data")` — variace
- Iter 6: `kb_search("systematic gaps in tracing data due to race condition")` — **identický** query (3. opakování)

Model po splnění úkolů (create_background_task, dispatch_coding_agent) se "vrátí" k hledání v KB, místo aby odpověděl. Patrně proto, že uživatel řekl "ulož do KB" ale storage tools byly odebrány — model nemůže splnit instrukci a opakovaně hledá.

### 3.3 Drift detection nedetekuje tento pattern

**Fakta z kódu** (`handler.py:610-644`):

Drift detection má 3 signály:

1. **consecutive_same >= 2** — vyžaduje 2× za sebou *identický* tool+args. Ale model střídá:
   - iter 1: `kb_search("systematic...")` → consecutive=1
   - iter 2: `create_background_task(...)` → consecutive=1 (reset!)
   - iter 3: `dispatch_coding_agent(...)` → consecutive=1 (reset!)
   - iter 4: `kb_search("systematic...")` → consecutive=1 (reset! jiný než iter 3)
   - iter 5: `kb_search("race condition...")` → consecutive=1 (reset! jiný args)
   - iter 6: `kb_search("systematic...")` → consecutive=1 (reset! jiný než iter 5)
   - **Nikdy nedosáhne 2**, protože args se střídají

2. **Domain drift** (3 iterace s 3+ různými doménami, žádná společná) — Domain history: `[search, task, task, search, search, search]`. Posledních 3: vše "search" → common domain existuje → nedetekuje.

3. **Excessive tools** (8+ distinct tools po 4+ iteracích) — Jen 3 distinct tools (kb_search, create_background_task, dispatch_coding_agent) → netriggerne.

**TOOL_DOMAINS** mapping (`tools.py:359`): `kb_search` → "search", `create_background_task` → "task", `dispatch_coding_agent` → "task"

### 3.4 MAX_ITERATIONS=6 pro long messages

**Fakta:**
- `MAX_ITERATIONS = 6` v `handler.py:50`
- S 39k base tokeny + 200-800 tokenů/iterace: po 4 iteracích překročí 40k → tier escalation
- Model vyčerpá 6/6 iterací (drift nedetekuje) → forced response
- Každá iterace na local_large trvá 186-309s, na local_xlarge 404-630s
- Typické use case: 1-2 tool calls stačí, 3 pro multi-intent, 6 je přehnané pro 126k zprávu

### 3.5 active=2 (dedup nefunguje kompletně)

**Fakta z kódu** (`main.py:262-315`):
- Dedup logika existuje: `_active_chat_stops` dict, `existing_event.set()` na řádku 289
- Ale `handle_chat()` kontroluje `disconnect_event.is_set()` pouze na začátku iterace (`handler.py:264-265`)
- Mezi iteracemi může být 300-600s! → druhý request se spustí paralelně
- Server (Kotlin) počítá `active` při `FOREGROUND_CHAT_START` nezávisle na Python dedup
- Otázka: posílá Kotlin/kRPC request 2× se stejným session_id? Nebo s různým?

### 3.6 GPU contention: embedding + 30b model souběžně

**Fakta z router logu:**
- `qwen3-embedding:8b` a `qwen3-coder-tool:30b` jsou nahrané na P40 současně
- CRITICAL chat request přijde na GPU, ale VRAM je obsazená oběma modely
- Router čeká 60s na dokončení aktivních requestů, pak force-unloadne embedding
- Embedding request (`c33347c6`) trval **129s** — zabíral GPU slot
- KV cache pro 40k + oba modely: 17GB + 4.5GB + 4GB = 25.5GB > 24GB VRAM → CPU spill i na 40k!

## 5. Návrh oprav (směr, ne konkrétní implementace)

### 5.1 Tier gap (P0)
Mezi 40k a 131k chybí mezivrstva. Buď:
- Přidat tier (např. 48k nebo 64k) s přijatelnou degradací
- Nebo zvýšit LOCAL_LARGE boundary zpět blíž k VRAM limitu
- Nebo clampnout tier selection: chat foreground nikdy nejde nad local_large (ořezat zprávu místo escalation)

Klíčová otázka: co je lepší — mírný CPU spill na 48k, nebo ořezání zprávy na 40k? Na P40 s 30b:
- 40k ctx: vejde se do VRAM ✓ (17+4=21GB)
- 48k ctx: hraničně (17+5≈22GB), mírný spill
- 64k ctx: spill ~3GB → degradace ale ne katastrofa
- 131k ctx: spill ~6GB → katastrofálně pomalé

### 5.2 Opakované tool calls (P1)
Model volá kb_search 4× se stejným/podobným query. Detekce musí zachytit:
- Tool+args již volán v historii (nejen consecutive)
- Varianty stejného query (fuzzy matching? nebo prostě "same tool name 3×")
- Model nedokáže splnit instrukci (storage tools odebrány, ale user chce ukládání)

### 5.3 Dynamický MAX_ITERATIONS (P1)
Pro dlouhé zprávy (>8k chars) je 6 iterací příliš — každá stojí 3-5 min. Snížit na 2-3, protože:
- Dlouhá zpráva = velký kontext = pomalé iterace
- Kontext roste s každou iterací → tier escalation risk

### 5.4 Dedup active=2 (P1)
Dedup logika v main.py existuje ale:
- `disconnect_event.is_set()` se kontroluje jen na začátku iterace (řádek 264)
- Pokud iterace trvá 300s, starý stream běží paralelně s novým
- Potřeba: zjistit zda Kotlin posílá 2× se stejným session_id, a zda Python korektně terminuje starý stream

### 5.5 GPU contention s embedding modelem (P2)
Embedding model na GPU zabírá VRAM a brání chat requestu mít plný VRAM pro KV cache. Řešení záleží na architektuře routeru — preemptive unload pro CRITICAL requesty, nebo dedikovaný embedding endpoint.

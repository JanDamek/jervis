## 3. Problémy k řešení

### 3.1 BUG: Summarizer timeout — GPU cleanup trvá déle než summarizer timeout (P0)

**Fakta:**
- Summarizer (`_summarize_long_message()` v `handler.py:1276-1345`) má timeout **30s** (`asyncio.wait_for(timeout=30.0)`)
- Summarizer používá `tier=ModelTier.LOCAL_FAST` (8k kontextu) s `X-Ollama-Priority: 0` (CRITICAL)
- Router přijme CRITICAL request ale GPU má nahrané **dva modely**: `qwen3-coder-tool:30b` + `qwen3-embedding:8b`
- Router musí embedding unloadnout, ale čeká na dokončení aktivních requestů
- Router čeká **60s** (wait loop s 2s intervalem) než force-unloadne embedding
- Summarizer timeout (30s) < Router GPU cleanup (60s) → **summarizer VŽDY timeoutne** když je embedding nahrán

**Chronologie problému:**
1. 22:25:01 — Python pošle summarizer request (CRITICAL)
2. 22:25:02 — Router ho přijme, zjistí že GPU má embedding + 30b, potřebuje unloadnout
3. 22:25:02 — Router začne čekat na 3 aktivní requesty (embedding + 2 NORMAL requesty)
4. 22:25:15 — KB embedding hotov (24s), ale stále 1 aktivní NORMAL request
5. 22:25:31 — **Python timeout (30s)** → summarizer fallback → zpráva se jen ořeže (ztráta 75% obsahu)
6. 22:26:02 — Router konečně force-unloadne embedding (60s) → ale Python dávno pokračuje
7. Decompose classifier (15s timeout) také nestihne → dvojité selhání

**Proč je embedding na GPU problém:**
- KB enrichment/ingestion požadavky (NORMAL priority) používají embedding model
- Embedding model zabírá ~4.5GB VRAM
- Router nerozlišuje "uvolni embedding HNED pro CRITICAL" vs "počkej na NORMAL"
- Dokonce i CRITICAL chat request musí čekat 60s na GPU cleanup

**Dopad:**
- Summarizer nikdy neproběhne dokud embedding běží na GPU
- 126k zpráva se jen pre-trimne na ~40k tokenů → ztráta 75% informací
- Agentic loop pracuje s ořezanou zprávou → nekvalitní výsledky
- Tento pattern se opakuje pokaždé když embedding model je nahrán (tj. téměř vždy — KB indexing běží neustále)

**Řešení (směr):**
1. **Router: preemptive unload pro CRITICAL** — Pokud CRITICAL request potřebuje GPU a embedding blokuje, okamžitě force-unloadnout (ne čekat 60s). CRITICAL = "teď, ne za minutu."
2. **Delší summarizer timeout** — Zvýšit z 30s na 90s (musí být > 60s router wait). Ale to jen maskuje problém.
3. **Dedicated embedding endpoint** — Embedding model na separátním CPU/GPU, nikdy neblokuje hlavní GPU.
4. **Kombinace**: Router fast-preempt (1) + summarizer timeout 90s jako safety net (2).

### 3.2 BUG: Background offload suggestion závisí na úspěchu summarizeru (P1)

**Fakta:**
- `handler.py:198-204`: Hint pro LLM "navrhni background task" se injektuje POUZE do summarized_content
- `handler.py:323-328`: FOCUS hint s background suggestion v `if is_summarized:` větvi
- Když summarizer selže (timeout), `is_summarized=False` → hinty se NIKDY neinjektují
- Model vidí ořezanou zprávu bez jakéhokoliv pokynu navrhnout background offload
- Uživatel explicitně řekl: "nezafungoval okamžitý dotaz, zda to neřešit jako background"

**Kód:**
```python
# handler.py:323-328 — background hint jen při is_summarized
if is_summarized:
    focus_hint += (
        " Pracuješ se SOUHRNEM originální zprávy. ..."
        "Pokud zpráva obsahuje příliš mnoho úkolů na zpracování v chatu, "
        "navrhni uživateli vytvoření background task pomocí create_background_task."
    )
```

**Řešení (směr):**
- Background offload suggestion musí být injektován pro VŠECHNY dlouhé zprávy (>16k chars), nezávisle na summarizer úspěchu
- Buď vždy připojit FOCUS hint s background suggestion pro `msg_len > SUMMARIZE_THRESHOLD`
- Nebo lépe: Pokud summarizer selže, IHNED (bez čekání na agentic loop) navrhnout background task:
  ```
  "Zpráva je příliš dlouhá pro zpracování v reálném čase.
   Vytvořím background task pro podrobné zpracování — bude to rychlejší a důkladnější."
  ```
- Model by měl mít jasnou instrukci v system prompt: pro zprávy >16k chars VŽDY navrhnout background jako první krok

### 3.3 BUG: Paralelní zpracování zpráv — dedup nefunguje (P1)

#### A) Stejná zpráva → active=3 (SSE reconnect/retry)

**Fakta:**
- Test #2 měl `active=2`, test #3 má `active=3` — situace se zhoršuje
- Server log: `FOREGROUND_CHAT_START: active=3`
- Tři souběžné SSE streamy na stejnou konverzaci!
- Dedup v `main.py:262-315` existuje — `disconnect_event.set()` na řádku ~289
- Ale `handle_chat()` kontroluje `disconnect_event.is_set()` POUZE na začátku iterace (`handler.py:341`)
- Mezi iteracemi je 268-300s! → starý stream běží paralelně
- Kotlin server patrně posílá request vícekrát (timeout → retry? SSE reconnect?)

#### B) Dvě RŮZNÉ zprávy → obě zpracovávají paralelně (UI screenshot 23:58-23:59)

**Fakta z UI:**
- 23:58 — uživatel posílá: "potřebuji aby jsi prošel architekturu projektu a to ve větvi develop."
- Chat začne zpracovávat → "Analyzuji výsledky..." (spinner)
- 23:59 — uživatel posílá follow-up: "Je třeba aby to bylo v KB"
- Chat začne zpracovávat DRUHOU zprávu → DALŠÍ "Analyzuji výsledky..." (spinner)
- **DVĚ zprávy se zpracovávají PARALELNĚ** — obě mají svůj spinner v UI

**Proč je to špatně:**
- Druhá zpráva je follow-up k první ("aby to bylo v KB" → doplnění kontextu ke "prošel architekturu")
- Měly by se zpracovat SEKVENČNĚ: první zpráva dokončí, druhá se zpracuje s plným kontextem
- Nebo: první zpráva se zruší a obě se spojí do jednoho požadavku
- Aktuálně: obě běží nezávisle na GPU → 2× GPU load, 2× pomalé, výsledky nekonzistentní

**Očekávané chování:**
1. **Varianta A (queue):** Druhá zpráva ČEKÁ dokud první nedokončí. UI zobrazí "Čeká na dokončení předchozí zprávy..." (ne druhý spinner)
2. **Varianta B (cancel+merge):** První zpracování se zruší, obě zprávy se zpracují dohromady (merge kontext)
3. **Varianta C (cancel+new):** První zpracování se zruší (disconnect_event), druhá zpráva se zpracuje sama (má kontext z conversation history)

**Dopad obou variant (A+B):**
- 2-3 paralelní LLM requesty = 2-3× GPU load (i když starší jsou "zombie")
- Možné konflikty při save_message do MongoDB (sequence collision)
- Zbytečné GPU cykly — GPU obsluhuje mrtvé requesty místo aktivního
- UX: uživatel vidí dva spinnery, neví který z nich je relevantní

**Řešení (směr):**
- **Kotlin strana:** Při odeslání nové zprávy VŽDY zavolat `POST /chat/stop` na předchozí stream, pak teprve poslat novou zprávu
- **Python strana:** Handler musí kontrolovat `disconnect_event` UVNITŘ tool execution a LLM call, ne jen mezi iteracemi
- `asyncio.Event` check v `_blocking_completion` (provider.py) jako cooperativní cancel
- **UI strana:** Zobrazit pouze JEDEN spinner (nejnovější zpráva). Starší zpracování buď skrýt nebo zobrazit "Zrušeno"
- Starý stream musí být ukončen OKAMŽITĚ, ne po 300s čekání na další iteraci

## 6. KRITICKÉ ARCHITEKTONICKÉ PRAVIDLO: Zákaz ořezávání zpráv

**Ořezávání (pre-trim/truncation) zpráv je NEPŘÍPUSTNÉ.**

Aktuální kód v `provider.py:_trim_messages_for_context()` ořeže zprávu na velikost kontextového okna.
Pro 126k zprávu s 40k kontextem to znamená ztrátu **75% obsahu**. To je fundamentálně špatný přístup.

**Správný přístup:**
1. **Sumarizovat** — LLM vytvoří strukturovaný souhrn zachovávající VŠECHNY požadavky
2. **Background task** — Pro zprávy s desítkami/stovkami úkolů: vytvořit background task
3. **Uložit do KB** — Originální zpráva se uloží do KB, agent se k ní může vrátit přes `kb_search`

**Pravidla:**
- **NIKDY** neořezávat scope, kontext, výsledky tool calls, uživatelské požadavky
- Agent MUSÍ mít vždy možnost vrátit se ke kterémukoliv výsledku
- KB read je relativně rychlé → agent se může zeptat znova (ne třeba cachovat)
- Jediné co je přípustné zkrátit: UI log/progress info pro zobrazení

**Co to znamená pro implementaci:**
- `_trim_messages_for_context()` v `provider.py` NESMÍ být fallback pro dlouhé zprávy
- Pokud summarizer selže → navrhnout background task, NE ořezat
- Pre-trim je akceptabilní POUZE pro conversation history (staré zprávy), NIKDY pro aktuální user message
- Aktuální user message musí být vždy zachována kompletně (v KB pokud ne v kontextu)

**Dokumentace:** Toto pravidlo je zapsáno v `docs/architecture.md` §16 "Long Message Processing — Pravidla".

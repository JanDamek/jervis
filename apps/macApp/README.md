# Jervis macApp ‚Äî native APNs helper for Compose Desktop

Two-app architecture on macOS:

- **macApp.app** (this folder) ‚Äî menubar-only Swift helper
  (`LSUIElement`), holds the `aps-environment` entitlement, registers
  for APNs, and relays the token + incoming push payloads to the JVM
  over `/tmp/jervis-macapp-apns.sock`.
- **Jervis.app** (existing `apps/desktop` jpackage bundle) ‚Äî Compose
  Desktop JVM. Dials the Unix socket on startup, feeds the APNs token
  into `registerTokenIfNeeded(platform="macos")`.

The two apps run independently. macApp is a background daemon (no Dock
icon); the Compose window is the normal user-facing UI started via
`./gradlew :apps:desktop:runPublic`. JVM embedding inside macApp.app
was attempted but abandoned: jpackage launchers hard-code their
`.cfg` path relative to the enclosing `.app`, so they can't be nested.

Windows and Linux Desktop builds keep using the bare JVM Compose
distribution with no background push ‚Äî see
`project-win-linux-desktop-push-deferred.md`.

`Info.plist` and `macApp.entitlements` are **generated from
`project.yml`** on every `xcodegen generate` run. Local linters /
editors reset them to minimal stubs ‚Äî that's fine, the full content
lives in the YAML spec. Do not rely on their on-disk contents.

No `.xcodeproj` is committed; regenerate it after edits to
`project.yml`:

```bash
( cd apps/macApp && xcodegen generate )
```

## Build + launch

Two apps, two runners ‚Äî mirrors how `apps/iosApp` is handled:

**1. macApp helper**

Pick one:

- **From IntelliJ** ‚Äî use the shared run config `macApp (Xcode)`.
  It runs `xcodegen generate` ‚Üí `xcodebuild -allowProvisioningUpdates
  build` ‚Üí relaunches the `.app` (menu-bar only). No Xcode GUI
  required.
- **From Xcode** ‚Äî `open apps/macApp/macApp.xcodeproj`, pick the
  `macApp` scheme, ‚åòR. Shared `.xcscheme` is generated by xcodegen,
  so it shows up the first time you open the project.

Xcode takes care of signing + provisioning once `Signing &
Capabilities` is configured (Team = GR74KSG8M9, see below). The
helper has `LSUIElement=true`, so it never shows up in the Dock.

**2. Compose Desktop client**

Shared IntelliJ run config `Desktop (runPublic)`, or CLI:

```bash
./gradlew :apps:desktop:runPublic
```

The JVM dials `/tmp/jervis-macapp-apns.sock` automatically (see
`PushTokenRegistrar.jvm.kt`) ‚Äî no env var required when the helper
is already running. Launch order doesn't matter:
`MacAppSocketBridge` retries until the socket appears.

Order doesn't matter: `MacAppSocketBridge` retries until the socket
appears, so you can start either process first.

Logs:

```bash
log stream --predicate 'subsystem == "com.jervis.macApp"' --style compact
```

## Enabling APNs (first-run signing setup)

`project.yml` includes `aps-environment=development` and hardcodes
`DEVELOPMENT_TEAM=GR74KSG8M9` (same team as `apps/iosApp` ‚Äî the
existing `.p8` APNs Auth Key signs both). `run-mac.sh` passes
`-allowProvisioningUpdates` so Xcode will auto-register the App ID
`com.jervis.macApp` with Push Notifications capability on first run.

Prerequisites (once per Apple ID):

1. Accept the latest **Program License Agreement** at
   `https://developer.apple.com/account` ‚Üí Agreements, Tax, and
   Banking. Without this, `xcodebuild` rejects every signing attempt
   with "PLA Update available".
2. Your Apple ID must be a member of Team `GR74KSG8M9` (or override
   `DEVELOPMENT_TEAM` in `project.yml` to your own team).
3. Run `./gradlew :apps:desktop:runMac` ‚Äî Xcode will prompt for
   Keychain access, register the App ID, and produce a signed
   `Jervis.app` with APNs entitlement.

Verify the token pipeline:

```bash
codesign -d --entitlements - build/macapp/Build/Products/Debug/Jervis.app
# should include <key>aps-environment</key> <string>development</string>
log stream --predicate 'process == "Jervis"' --style compact | grep APNs
# should log "APNs token: xxxxxxxx... device: <UUID>"
```

## IPC protocol

Unix socket at `/tmp/jervis-macapp-apns.sock`. One JSON message per
line, written by Swift, read by the JVM
(`MacAppSocketBridge.kt`):

```
{"kind":"token","hexToken":"...","deviceId":"IOPlatformUUID"}
{"kind":"payload","userInfo":{...}}
```

The JVM side uses `JERVIS_MACAPP_SOCKET` env (the macApp Swift host
never launches the JVM itself ‚Äî it's set manually or by a
LaunchAgent). If not set, the socket path defaults to
`/tmp/jervis-macapp-apns.sock` inside
`PushTokenRegistrar.jvm.kt`.
